# TCP/IP 四层与 OSI 七层

## 核心结论

- 实际工程中常用的是 TCP/IP 四层模型：网络接口层、网络层、传输层、应用层。
- OSI 七层模型更偏理论，5/6/7 层（会话/表示/应用）在实现里通常合并为"应用层"。
- 交换机主要工作在二层（数据链路 / 网络接口层），根据 MAC 地址转发帧；路由器工作在三层（网络层），根据 IP 地址转发包。
- TCP 通过三次握手建立连接，通过四次挥手（有时压缩成三个报文）可靠关闭连接，主动关闭方进入 TIME_WAIT。
- Nginx 可以既做七层（HTTP）代理，又做四层（TCP/UDP）代理，分别对应应用层网关和传输层网关。

---

## 1. TCP/IP 四层模型概览

TCP/IP 四层模型（自下而上）：

1. **网络接口层**（对应 OSI 物理层 + 数据链路层）
   - 典型协议 / 技术：Ethernet、Wi‑Fi、PPP
   - 单位：比特 / 帧
   - 关键点：MAC 地址、局域网内转发

2. **网络层**
   - 典型协议：IP、ICMP、路由协议（OSPF 等）
   - 单位：包（Packet）
   - 关键点：IP 寻址、跨网段路由

3. **传输层**
   - 典型协议：TCP、UDP
   - 单位：段（Segment, TCP）/ 数据报（Datagram, UDP）
   - 关键点：端口号、端到端连接、可靠/不可靠传输

4. **应用层**
   - 典型协议：HTTP、HTTPS、FTP、SMTP、DNS、SSH 等
   - 单位：报文（Message）
   - 关键点：面向具体业务的协议语义

与本知识库中关于 HTTP/SSE 的内容关联：HTTP/SSE 等都属于应用层协议，可参考 [LLM 流式输出与 SSE](../../agent/engineering/streaming-and-sse.md)。

## 2. OSI 七层模型与 5/6/7 的合并

OSI 七层（自下而上）：

1. 物理层：比特传输（电信号/光信号），网线、光纤等。
2. 数据链路层：在相邻节点之间可靠传输帧，使用 MAC 地址（交换机）。
3. 网络层：跨网络路由和逻辑寻址，使用 IP 地址（路由器）。
4. 传输层：端到端连接和端口寻址，TCP/UDP。
5. 会话层：建立/管理/终止会话（会话恢复、对话控制）。
6. 表示层：数据格式转换、加密解密、压缩解压（TLS、编码）。
7. 应用层：HTTP、FTP、DNS 等具体应用协议。

**现实中的合并：**

- 大部分操作系统和协议栈并不会把 5/6/7 层拆得这么细，而是统称为"应用层"。
- 会话管理（如 HTTP Keep-Alive、TLS 会话恢复）、数据表示（JSON/Protobuf、字符编码）、应用语义（REST、SQL）通常在同一个库或服务中一起实现。
- 因此在 TCP/IP 模型中，我们一般说"四层"，其中最上层就是"应用层（包含 OSI 的 5/6/7 功能）"。

## 3. 交换机 vs 路由器

| 设备 | 所在层（OSI/TCP-IP） | 寻址依据 | 主要作用 |
|------|----------------------|----------|----------|
| 交换机 | OSI 二层 / TCP-IP 网络接口层 | MAC 地址 | 在同一二层网络内转发帧，维护 MAC 地址表 |
| 路由器 | OSI 三层 / TCP-IP 网络层 | IP 地址 | 在不同网络之间转发 IP 包，进行路由选择 |

类比：

- 交换机像"同一栋大楼内的楼层管理员"：只在楼内送货，记的是房间号（MAC）。
- 路由器像"城市级的邮局"：看的是城市+街道+门牌号（IP），决定包裹该走哪条跨城路线。

数据路径上，一般是：

> 主机 → 交换机（局域网内） → 路由器（网关） → 若干路由器 → 目标网段交换机 → 目标主机

只有在二层/三层能正常寻址的前提下，传输层（TCP）和应用层（HTTP 等）才有存在的意义。

## 4. TCP 三次握手与四/三次挥手

### 4.1 三次握手（建立连接）

假设客户端发起连接：

1. **SYN**：客户端 → 服务器，`SYN=1, seq=x`。
2. **SYN-ACK**：服务器 → 客户端，`SYN=1, ACK=1, seq=y, ack=x+1`。
3. **ACK**：客户端 → 服务器，`ACK=1, seq=x+1, ack=y+1`。

随后双方进入 ESTABLISHED 状态，可以双向收发数据。

### 4.2 四次挥手（关闭连接）

以客户端主动关闭为例：

1. 客户端 → 服务器：`FIN=1, seq=u`（客户端进入 FIN-WAIT-1）。
2. 服务器 → 客户端：`ACK=1, seq=v, ack=u+1`（服务器进入 CLOSE-WAIT，客户端进入 FIN-WAIT-2）。
3. 服务器（应用层处理完剩余数据后）→ 客户端：`FIN=1, seq=w, ack=u+1`（服务器进入 LAST-ACK）。
4. 客户端 → 服务器：`ACK=1, seq=u+1, ack=w+1`，然后客户端进入 **TIME_WAIT**，等待 2MSL 后真正关闭。

### 4.3 为什么有时看起来只有"三次挥手"？

如果服务器在收到客户端的 FIN 时：

- 已经没有数据要发送；
- 应用层也立刻决定关闭连接；

那么服务器可以把"对客户端 FIN 的 ACK"和"自己的 FIN"合并到同一个报文中发送：

1. 客户端 → 服务器：FIN
2. 服务器 → 客户端：ACK + FIN（合并在一个包内）
3. 客户端 → 服务器：ACK（客户端进入 TIME_WAIT）

逻辑上仍然是"四次挥手"的四个步骤，只是 2/3 步合并成了一个包，从抓包上看是三个报文。

### 4.4 为什么需要 TIME_WAIT？

主动关闭的一方在发送最后一个 ACK 后进入 TIME_WAIT 状态，主要原因：

1. **确保连接可靠关闭**：
   - 如果最后一个 ACK 丢包，服务器会重传 FIN。
   - 处于 TIME_WAIT 的一方可以再次回复 ACK，保证对端最终能正常关闭。

2. **避免旧连接报文污染新连接**：
   - 相同四元组（源 IP/源端口/目标 IP/目标端口）的新连接可能在一段时间后复用。
   - TIME_WAIT 持续 2MSL，可让旧连接残留的报文在网络中自然消失，避免被新连接误收。

因此：

- 谁主动发起关闭，谁通常进入 TIME_WAIT。
- 在短连接、服务器主动关闭的场景（如 HTTP/1.0，无 keep-alive）下，服务器上可能堆积大量 TIME_WAIT 连接。

## 5. Nginx 在分层中的位置

Nginx 是一个多层代理服务器：

1. **作为七层（应用层）网关**：
   - 使用 `http` 模块解析 HTTP/HTTPS 协议。
   - 可以基于 URL、Host、Header、Cookie 等做路由和负载均衡。
   - 典型配置：反向代理 Web 服务、做静态文件服务、TLS 终止。

2. **作为四层（传输层）网关**：
   - 使用 `stream` 模块，仅转发 TCP/UDP 流，不解析上层协议。
   - 可以为 MySQL、Redis、SMTP 等做 TCP 负载均衡。

对应关系：

- 在 **OSI 模型**里：
  - http 模块 ≈ 七层负载均衡（应用层）。
  - stream 模块 ≈ 四层负载均衡（传输层）。
- 在 **TCP/IP 四层模型**里：
  - HTTP 代理模式：工作在应用层（第 4 层）。
  - TCP 代理模式：工作在传输层（第 3 层）。

常见说法中的"L4 / L7 负载均衡"是按 OSI 模型编号来的：

- L4：基于 IP + 端口的转发（不看应用层内容）。
- L7：解析应用层协议，根据 URL/头部等做更细粒度路由。

## 6. 典型数据流：浏览器访问网站

整体路径可以串起来理解：

1. 应用层：浏览器构造 HTTP/HTTPS 请求。
2. 传输层：TCP 三次握手建立连接（客户端与服务器/Nginx）。
3. 网络层：根据目标 IP，数据包在多个路由器之间转发。
4. 数据链路层/物理层：每一跳上，通过交换机和物理链路传输帧/信号。
5. 应用层：服务器或 Nginx 收到请求，解析 HTTP，返回响应；之后通过 TCP 挥手关闭连接。

---

## 7. 易混点速记

- 理论七层 vs 实际四层：**5/6/7 在工程实践中几乎总是合并为"应用层"**。
- 交换机 vs 路由器：前者看 MAC、后者看 IP；前者主要局域网，后者跨网络。
- 三次握手 vs 四/三次挥手：
  - 握手一定是三次；
  - 挥手逻辑上是四次，但 ACK 和 FIN 可合并成一个包，所以抓包上可能看到"3 个报文"。
- TIME_WAIT：**主动关闭的一方**进入；目的是可靠关闭连接 + 清理旧报文。
- Nginx：既可以是 L4 负载均衡（stream），也可以是 L7 负载均衡（http），关键在于是否理解并操作应用层协议。
